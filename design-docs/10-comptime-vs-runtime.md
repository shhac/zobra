# 10 — Comptime vs runtime

## The tension

Cobra is **runtime**: you build the command tree imperatively in `init()`, register children at startup, and the whole tree lives behind a heap pointer. This works because Go has GC and reflection (`reflect.StructOf`, `reflect.Value.FieldByName`) to introspect arbitrary structs at runtime.

Zig has neither GC nor runtime reflection. But Zig has **comptime**: arbitrary code that runs at compile time, with full type introspection (`@typeInfo`, `std.meta.fields`, `@field`). The tension is: do we mirror cobra's runtime registration model, or do we use Zig's comptime affordances to make the tree declarative?

The answer is **both**. Each path has a use case. We design the runtime layer first (it's the differential-test target and the cobra-port path) and add the comptime layer as a thin generator on top once the runtime is stable.

## The runtime path (Phase 1–6)

The default. Mirrors cobra closely. Every example in [02-cobra-mapping.md](02-cobra-mapping.md) uses this form.

```zig
const root = try zobra.Command.init(allocator, .{
    .use = "myapp",
    .short = "my CLI",
});
defer root.deinit();

var name: []const u8 = "world";
try root.persistentFlags().stringVarP(&name, "name", "n", "world", "who to greet");

const greet = try zobra.Command.init(allocator, .{
    .use = "greet [target]",
    .short = "Print a greeting",
    .args = zobra.args.maximumN(1),
    .run_e = greetRun,
});
try root.addCommand(greet);

try root.execute(allocator, std.os.argv);
```

**Strengths**:

- 1:1 mental model with cobra.
- Dynamic — children can be added at runtime, conditional on flags or environment.
- Easy to test in isolation; each piece is just a value.
- Plays well with allocators — every allocation has a clear free path.

**Weaknesses**:

- Verbose — six or eight `try`s per Command.
- Errors are runtime — typos in flag names surface during the first `execute`, not at compile time.
- No compile-time validation of flag-group references (`markFlagsMutuallyExclusive("a", "b")` doesn't check that "a" and "b" exist).

## The comptime path (Phase 7, optional)

A declarative alternative. The user writes a struct literal at comptime; zobra walks it with `@typeInfo` and synthesises the runtime tree.

```zig
const Spec = struct {
    pub const command = .{
        .use = "myapp",
        .short = "my CLI",
        .persistent_flags = .{
            .name = .{ .type = []const u8, .short = 'n', .default = "world", .desc = "who to greet" },
        },
        .children = .{
            .{
                .use = "greet [target]",
                .short = "Print a greeting",
                .args = .max(1),
                .run = greetRun,
            },
        },
    };
};

pub fn main(init: std.process.Init) !void {
    const arena = init.arena.allocator();
    var cli = try zobra.declare(arena, Spec);
    defer cli.deinit();
    try cli.execute(arena, try init.minimal.args.toSlice(arena));
}
```

`zobra.declare` is a comptime function: it walks `Spec.command` with `@typeInfo`, generates the equivalent runtime calls, and returns a `*Command` ready to execute.

**Strengths**:

- Declarative — reads more like a config than a program.
- **Compile-time validation** — `markFlagsMutuallyExclusive("a", "b")` becomes a comptime check that "a" and "b" appear in the flag spec; typos fail to compile. The flag-group constraint sites live next to the flag definitions; the compiler enforces consistency.
- Zero runtime cost beyond the synthesised allocations (which are exactly what the runtime form would do anyway).

**Weaknesses**:

- Less dynamic — children must be known at comptime. Plugins / runtime-loaded subcommands don't fit.
- Steeper learning curve for users new to Zig comptime.
- Compile-time error messages from `@typeInfo` walks can be tricky.

## How they interact

The comptime form is a **generator** for the runtime form. It produces the same runtime structures: a `*Command` with the same `FlagSet`, `children` list, hook table. After `zobra.declare(...)` returns, the resulting `Command` is indistinguishable from one built by hand — same `deinit`, same `execute`, same observable behaviour.

This means:

- The differential test suite uses the runtime path. It doesn't need to test the comptime path separately, because the comptime path produces the same runtime structures.
- A consumer can mix forms: declare the bulk of the tree at comptime, then add a runtime-loaded child via `cli.addCommand(...)`.

## Where comptime is used internally (Phase 1+)

Independent of the comptime *user* path, zobra uses comptime internally where it clarifies the implementation:

- **Flag-type registration**: the 16 scalar `*VarP` methods are generated by a comptime `inline for (...)` over a flag-type table, rather than copy-pasted. One source of truth; one place to edit.
- **Args-validator combinators**: `matchAll(.{ a, b, c })` builds a struct of validators at comptime and dispatches at runtime.
- **Token-tag dispatch**: the parser's `switch (token)` is exhaustive (Zig's `union(enum)` requires it); the compiler enforces every token kind is handled.

These are comptime *implementation* uses; they're invisible to the user and don't affect the API shape.

## Decision: which form to ship first

**Phase 1–6 ships the runtime path only.** The comptime path is Phase 7 — it depends on the runtime form being stable first (since it generates that form), and the differential test suite needs the runtime path anyway.

If a consumer is ready to use zobra before Phase 7, they use the runtime path. If they prefer the comptime form, they wait or land it themselves.

## What about cobra's `init()` registration pattern?

In cobra, you typically write:

```go
var rootCmd = &cobra.Command{...}

func init() {
    rootCmd.AddCommand(greetCmd)
    rootCmd.PersistentFlags().StringVarP(&name, "name", "n", "world", "who to greet")
}
```

Zig has no `init()` side-effect registration. A direct port becomes:

```zig
pub fn main(init: std.process.Init) !void {
    const arena = init.arena.allocator();
    const root = try buildRoot(arena);
    defer root.deinit();
    try root.execute(arena, try init.minimal.args.toSlice(arena));
}

fn buildRoot(allocator: Allocator) !*zobra.Command {
    var root = try zobra.Command.init(allocator, .{ .use = "myapp" });
    errdefer root.deinit();
    try root.addCommand(greetCmd);
    try root.persistentFlags().stringVarP(...);
    return root;
}
```

The `init()` function becomes a `buildRoot` function. The mental model is the same; the call site moves from "package import time" to "main entry."

## Open questions

- **Does the comptime form support inheritance of persistent flags through nested struct types?** Probably yes — we can encode `.persistent_flags = .{...}` on any node and have `declare` propagate. Defer until Phase 7.
- **Should comptime spec validation produce useful error messages?** Zig 0.14+ improved `@compileError` ergonomics; we'll use it. The target is "if a user mistypes a flag name in `mark_flags_mutually_exclusive`, the compile error names the missing flag and the location."
